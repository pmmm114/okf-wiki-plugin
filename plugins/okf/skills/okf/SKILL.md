# OKF 번들 스킬

## 1. 트리거 조건

repo에 `.okf/` 번들(또는 `.okf-wiki.json`이 가리키는 번들)이 있고, 다음 중
하나일 때 이 스킬을 쓴다: 지식 캡처·개념 작성 요청 / 문서화된 자산 변경에
따른 번들 갱신 / 번들 검증·색인 / 번들 지식으로 답해야 하는 질문. 번들이
없는 repo의 일반 코딩 작업에는 개입하지 않는다.

## 2. 작성 플로우

1. 루트 index를 먼저 읽고 새 개념의 배치 위치와 이웃 개념을 파악한다.
2. 배치 판단: 개념은 주제 그룹(하위 디렉토리) 단위로 묶는다(스펙 §3).
   새 개념이 기존 디렉토리의 주제와 다르면 하위 디렉토리를 새로 만들고,
   한 디렉토리에 이질적인 주제가 쌓여 있으면 분할을 제안한다 — 기존
   디렉토리에 몰아넣는 것을 기본값으로 삼지 않는다. 루트 상대 링크는
   문서 이동에도 깨지지 않으며, 이동 후 `okf index <번들경로> --write`로
   각 디렉토리 index를 재생성한다.
3. 개념 생성 규칙: frontmatter `type` 필수(비어있지 않게), `description`
   1문장, 기존 문서로부터 백링크 ≥1(그룹 경계와 무관하게 루트 상대
   링크로), 본문은 설명이 아니라 **답**(스키마 컬럼·조인 키·명령어·수치)을
   우선 담는다. 한 파일 한 개념. 인식층을 표시하려면(선택) `layer`
   (정보/지식/지혜)를 부여하고, 상위 층이면 근거 하위 개념을 `derived_from`으로,
   정보 층이면 출처(`resource`·`# Citations`)를 매단다 — 정의·정초 순서·불변식은
   `reference/LAYERS.md`가 정본이다.
4. 같은 변경에서 index 항목과 가장 가까운 log.md 일자 엔트리를 추가한다.
5. 마치면 `okf validate <번들경로> --strict`를 실행하고 error·warn을 모두
   해소한 뒤 끝낸다.

## 3. 유지 플로우

문서화된 자산이 변경되면 해당 개념 문서를 갱신하고 가장 가까운 log.md에
`## YYYY-MM-DD` 엔트리를 남긴다(같은 변경 묶음에서). 깨진 링크는 오류가
아니라 미작성 지식의 신호다 — 필요하면 대상 개념을 새로 작성해 채운다.

## 4. 소비 플로우

- 컨텍스트에 `<okf-context>` 블록이 있으면 그것을 1차 정답 소스로 쓴다.
- 더 깊은 근거가 필요하면 번들의 해당 개념 파일을 직접 읽는다.
- 형식 규칙(frontmatter·index·log·컨포먼스) 판단이 필요하면
  `reference/SPEC.md`(OKF v0.1 원문)를 참조한다 — 스펙을 요약해 재기술하지
  말고 원문을 근거로 삼는다.
- 인식층 구분이 필요하면(정보·지식·지혜) `reference/LAYERS.md`를 정본으로
  따른다 — 판단은 지혜를, 이해는 지식을, 사실값은 정보를 우선하고, 판단을
  근거로 쓸 때는 그 근거 사슬(`derived_from`→출처)까지 확인한다.

## 5. 실행 위임

검증·색인·그래프·컨텍스트·로그 조작은 전부 `okf` 맨 명령에 위임한다:
`okf validate | index | graph | context | log append | init`. 이 스킬 안에 자체
스크립트를 두지 않는다.

## 6. 승격(study) 플로우

메모리·명시 요청을 지식 개념으로 올리는 흐름. `/study`(또는 "적재/승격해줘")가
트리거다. `.okf-study/inbox.md`의 후보를 **선택적으로** 승격한다(전체가 아니라 고른
부분집합 — 모호하면 물어본다).

- **후보 원천**: `capture: review/auto`면 inbox에 쌓인 것, `off`면 MEMORY.md·명시 요청.
- **선별**: 장기 지식(스키마·명령·결정·규약)만. 상호작용 취향·일회성은 제외 —
  남긴 후보는 아래 **개념화**에서 인식층으로 분류한다.
- **개념화**: §2 규칙대로 작성하고 **주제 하위디렉토리**에 배치한다. 카테고리는
  `type`(필수) + 주제 디렉토리 + **인식층 `layer`**이며, `tags`는 선택 메타일 뿐
  배치·필터 축이 아니다. **인식층 판정은 승격의 필수 단계**다 — 후보가 담은 인식
  고도를 판정해 `layer`를 부여하고, 상위 층이면 근거 하위 개념을 `derived_from`으로
  잇는다. 층 어휘·판정 휴리스틱·정초 순서는 `reference/LAYERS.md`가 정본(§1 정의·
  §5 저작 가이드) — 어휘를 재기술하지 말고 그 문서를 따른다. `<topic>`/`--type`
  인자로 후보를 그 주제/타입에 한정할 수 있다.
- **검증·드레인**: `okf validate --strict` 통과분만 원장에 promoted 기록하고 inbox에서
  제거한다. 버릴 후보는 discarded로 기록(동일 스니펫 재적재 방지).
- **디스패치**: 승격 개념을 소비처 핸들러로 넘긴다. 핸들러 실행은 **로컬 trust
  승인**이 필요하고, 미승인이면 개념은 승격되되 핸들러만 보류된다(`/study --trust`).
- 실행은 전부 `study.py`(list/resolve/clear/dispatch)·`study_trust.py`·`okf` CLI에
  위임한다 — 이 스킬은 판정만 한다(§5 원칙).
